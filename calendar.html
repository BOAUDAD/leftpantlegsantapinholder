Date & Time Picker

Usage

Picking dates

Many parts of our system rely on date-picking to create reports, set the start and end times for campaigns, and to filter table views. Date pickers can be used for selecting a single date or a date range, and should allow the user to easily select days without having to separately specify day, month, and year via a text input.

Formatting

Date

Expand
Expand

Within the current calendar year, display the date without the year. Otherwise, display the date and year.

Time

Expand

Time is displayed in the hh:mm format, where the first digit of the hour is only displayed if it is a number other than zero. AM and PM are uppercase with no periods, consistent with Material Design standards.

Combined date & time

Expand

In cases where both a date and time is required, we can minimize user effort by combining them in one field and using the combined date & time picker shown below.

Portrait Styles

The portrait date pickers have a 320px width and 480px height, which will prevent the need for scrolling on small phones.

Dropdowns appear upon tabbing to the input fields, which allows users with visual impairments to access the picker with only their keyboard (see the keyboard input examples below). Sighted users can achieve the same effect by double-clicking the input, which allows for more granular control over the time field. We use dropdowns to limit user input, making them less prone to error. These dropdowns should be disabled on mobile devices.

Dev notes

  • The dates should behave as though they have the .wave-effect class applied, but I wasn't able to coerce them to do that.
  • Ideally, more than one date selector can be shown when picking ranges. This was beyond my abilities so I left it as an exercise for the reader.
  • If a daterange (start and end date) is being selected, two datepickers should appear in the modal. The left picker should always be at most the previous month before the right picker, and conversely the right picker should always be at least the month after the one on the left.

Acceptance Criteria

User testing:

  1. Users should be able to select any arbitrary date that is permitted by the logic of the page.
  2. If dates in the past or some fixed time in the future are not valid selections, the interface should reflect these restrictions and users should be able to understand them.

Tech testing:

  1. If a date is required and a user attempts to proceed without selecting one, follow the text input error conventions.
  2. Any entered dates outside valid ISO-whatever ranges will default to today's date.
  3. Invalid dates are tooltipped according to the tooltip rules.
  4. Where datepickers are shown as modal dialogs, they should follow the dialog rules.

Accessibility testing:

  1. Keyboard-accessible dropdowns are triggered upon tabbing to modal inputs.
