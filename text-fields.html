Text fields

Styles & States


Normal & Focus

Text fields have a 1px bottom border until they are in focus, at which point the border becomes 3px tall. We do not use placeholders because users who skim forms before submitting can easily mistake placeholders for text inputs.

Type in the First Name field to see the interaction.

Warning

Disabled

Disabled fields and corresponding labels are at 26% opacity to indicate that they are not actionable.

Warning

Required

If some fields are required

Include an asterisk next to the required fields.


If all fields are required

Forgo the asterisk next to the label and include text above the form that indicates all fields must be completed.

Warning



Validation

Invalid

An invalid field shows a 3px orange border and a warning icon—visual cues that are sufficient to indicate an error to a colorblind person. When in focus, the invalid field also shows an intuitive error message.

Enter an incorrectly formatted email address to see the interaction.


Valid

Upon successful validation, a field returns to its normal state.

Hints

If applicable, hints appear below the field when the field is in focus.

Each hint disappears when it is satisfied.

Hints are a maximum of 2 lines to avoid overlapping the fields below.

Text areas


Text areas look and function similar to text input fields, but allow for multiple lines of text.

Since text areas grow horizontally to accomodate additional lines, they should be placed at the bottom of forms wherever it makes sense. This minimizes alignment problems with multi-column layouts.

Enter text to see the behavior.

Character counter



For optimal usability, let the user know where there are character restrictions by using the character counter, which appears on focus.

Warning

Labels


Standard labels

Use standard labels when there are many input fields on a screen, and on mobile phones, where real estate is limited.

Warning

Large labels

Use header labels for simple forms on large screens, where input fields are sparse. Header labels make optimal use of screen real estate, and they also allow users with screen readers to jump from header to header to hear an outline of the page's main content.

Label Accessibility

To ensure that users using screen readers can easily associate form labels with their corresponding input fields, add “for” attribute to label tags to match the ids on the corresponding form fields, as shown.

<div class="input-field"> <input id="campaign_name" type="text" class="validate"> <label for="campaign_name">Campaign Name</label> </div>

Layout

Fields should have plenty of vertical space between them to allow for easy touch on mobile devices. We recommend 44px between fields, and 44px between fields and their corresponding labels. Fields should be horizontally aligned for a clean UI.

Since text areas grow horizontally to accomodate additional lines, they should be placed at the bottom of forms wherever it makes sense. This minimizes alignment problems with multi-column layouts.


Acceptance Criteria

User testing:

  1. Users understand that input fields are input fields.
  2. Users understand which fields are required.
  3. Hints and error messages make sense to users.
  4. Most users do not become frustrated with the validation workflow.

Accessibility testing:

  1. Forms can be completed using only a keyboard.
  2. Users using screen readers can associate form labels with their input fields.

Tech testing:

  1. The normal, in focus, disabled, invalid states, as well as hints and labels, match the designs provided.
  2. Individual hints disappear once they are satisfied.
  3. All required fields are marked as shown.
  4. Empty required fields become invalid upon submit.
  5. Invalid entries for fields with specific rules (i.e. passwords) immediately display as invalid when the fields lose focus.
